# LunarMochiVoxへのコントリビューション

はじめに、LunarMochiVoxへのコントリビューションに興味を持っていただき、 ありがとうございます。
このドキュメントをお読み頂き、スムーズなコントリビューションへのご協力をお願いします。

## コントリビューションの方法

LunarMochiVoxへのコントリビューションは、以下の方法を通じて行うことができます：

### GitHubリポジトリへのPull Request

これが最も一般的かつ、一番に想像される方法でしょう。
以下は、Pull Requestを通じてコントリビューションする際の基本的な手順です：

1. **リポジトリをフォーク**: GitHub上でLunarMochiVoxのリポジトリを自分のアカウントにフォークします。
2. **ローカルにクローン**: フォークしたリポジトリを自分のローカル環境にクローンします。
3. **新しいブランチを作成**: 変更を加えるための新しいブランチを作成します。
4. **変更を加える**: コードの修正や新しい機能の追加など、必要な変更を行います。
5. **コミット**: 変更をコミットし、意味のあるコミットメッセージを付けます。
6. **プッシュ**: 変更を自分のリポジトリにプッシュします。
7. **Pull Requestを作成**: GitHub上で、LunarMochiVoxのリポジトリに対してPull Requestを作成します。変更内容や目的を明確に説明してください。この説明文に入力が推奨される情報は別途記述します。
8. **レビューとマージ**: コントリビューションがレビューされ、必要に応じてフィードバックが提供されます。最終的に、変更が承認されると、LunarMochiVoxのリポジトリにマージされます。

箇条書きのため、簡略化した手順等もございますが、基本的な流れは上記の通りです。

GitやGitHubの使用方法といった、基本的な知識に関しては、GitHubの公式ドキュメントや、その他のオンラインリソースを参照してください。
このリポジトリでは、GitHubの使用方法に関するサポートは提供しておりませんので、ご了承ください。

#### Pull Requestの説明文に入力が推奨される情報

Pull Requestの説明文には、以下の情報を入力することが必須です：

- **変更の目的**: どのような変更を行ったのか、なぜその変更が必要なのかを説明してください。
これは、取り組んだIssue番号等の関連する情報を記述することも含まれます。
- **サイドエフェクト**: この変更を適用するにあたり、影響を受け修正した箇所・影響を受ける可能性のある箇所など、サイドエフェクトに関する情報を提供してください。
例えば：インターフェイス `Parent` を実装する子クラス `Child` で オーバーライドしているメソッドシグニチャーを変更する必要があった場合等。
- **テスト**: 変更を適用するにあたり、どのようなテストを行ったのか、またはどのようなテストが必要であると考えているのかを説明してください。

また、以下の情報は必須ではありませんが、提供していただくことでレビューやマージのプロセスがスムーズになる可能性があります：
- **変更に関する注意点・補足情報**: 変更に関する注意点や、レビューアーが知っておくべき補足情報があれば提供してください。
例えば：コードベース以外での変更が必要な場合（例：ドキュメントの更新、依存関係の追加など）や、特定の環境でのみ発生する問題に対処するための変更など。
- **関連するリポジトリアイテム**: 変更に際して、関連、もしくはトリガーとなったアイテム（Issue, Pull Request, Discussion等）があれば、その情報を提供してください。
- **変更のスクリーンショットやGIF**: 変更がユーザーインターフェイスに影響を与える場合、変更前と変更後のスクリーンショットやGIFを提供することが推奨されます。これにより、レビューアーが変更の影響を視覚的に理解しやすくなるだけでなく、レビュアーが実際にテストする際にも役立ちます。

## コントリビューションのガイドライン

LunarMochiVoxへのコントリビューションに際して、以下のガイドラインを遵守してください：

### コードスタイル

原則として、[Google Java Style Guide](https://google.github.io/styleguide/javaguide.html)に従ってください。
ただし、以下の要素を例外とします：

- ワイルドカードインポート: テストコード (`/src/test/java/`以下) でのみ、ワイルドカードインポートを許可します。それ以外では、ワイルドカードインポートは使用しないでください。
- カラムリミット: 制限なし。現代のモニター環境やエディタ機能は充実しています。無理に改行を行う必要はありません。
ただし、オープンになることを前提とし、著しく可読性を損なうような記述は避けてください。
- 行ラップ: 4.5.1 `Where to Break` のルールのみを適用します。その他の制限は問われません。
- アノテーション: 同じ行に複数のアノテーションを記述しないでください。アノテーションは、クラス、メソッド、フィールドなどの宣言の前に、1行ずつ記述してください。
順番については特に規定しませんが、Style Guideの記述の通りドキュメントブロックの直後に配置してください。
- Javadoc: 7.3 `Where Javadoc is used` でのルールに加え、以下のルールを適用します。
  - 外部へ露出していないが、APIとしてコードベース内で広く使用されるクラス、メソッド、フィールドには、Javadocコメントを記述することが推奨されます。
  - Javadocコメントは、クラス、メソッド、フィールドなどの宣言の前に配置してください。

### 生成AIの使用について

全面的に許可します。
ただし、生成AIを使用して生成されたコード、ドキュメント等は、
必ずコントリビューター本人が内容を理解し、責任を持って管理してください。

Pull Requestの説明文やその他ドキュメントに対しても、生成AIを使用して生成された内容を含められますが、
同様に、コントリビューター本人が内容を理解し、必要があれば変更に沿うように修正してください。

上記の理由として、 **生成AIのみで作られたコンテンツは、とどのつまりAIの仕事であり、あなたの成果ではない** ためです。
「だれでも出来ること」ではなく、「あなたにしか出来ないこと」を、貢献の主題としてください。
